# Hybrid Writing Style

이 문서는 `topic-wiki-factory`에서 쓸 문체를 정리한다.

목표는 단순하다.

- 네 블로그 글의 장점인 `유머`, `공감`, `예시`, `생각나게 하는 비유`를 살린다
- 동시에 위키 문서에 필요한 `명확성`, `구조`, `검색성`, `재사용성`도 유지한다

즉 이 문체는
`코딩애플처럼 술술 읽히는데, 나중에 다시 찾기 쉬운 위키 문체`
를 지향한다.

## 이 문체가 지향하는 느낌

- 너무 딱딱한 설명서도 아니다
- 너무 감성적인 블로그도 아니다
- 읽을 때는 블로그처럼 잘 읽히고
- 다시 찾을 때는 위키처럼 바로 찾힌다

한 줄로 말하면:

> 친근하게 설명하되, 구조는 엄격하게 잡는다.

## 가져올 장점

### 블로그 쪽에서 가져올 것

- 도입부에서 사람을 끌어당기는 질문이나 장면
- 유머러스하지만 과하지 않은 말맛
- 독자가 실제로 겪는 상황 예시
- "아 이거 내 얘긴데?" 싶은 공감 포인트
- 마지막에 생각이 남는 한 문장

### 리서치 쪽에서 가져올 것

- 중요한 답을 앞에 두는 구조
- 짧은 문장, 짧은 문단, 한 문단 한 생각
- 능동태, 쉬운 단어, 설명형 소제목
- 스캔 가능한 리스트와 요약
- 장면 뒤에는 바로 원칙을 붙이는 방식

## 기본 톤

### 1. 친근하지만 가볍게 흘리지 않는다

- 반말 감각이나 구어체는 살릴 수 있다
- 하지만 문서 전체가 농담으로 흩어지면 안 된다
- 웃기려고 쓰는 게 아니라, 덜 겁먹게 하려고 쓴다

좋은 예:

- "여기서 사람 확인이 없으면 월요일이 사고 수습으로 시작될 수 있다."
- "승인 버튼을 많이 단다고 안전해지는 건 아니다. 회의만 늘어난다."

피할 예:

- "이건 진짜 미쳤다"
- "레전드다"
- "개꿀"

### 2. 독자의 막힘을 먼저 안다

- 설명보다 먼저 "어디서 막히는지"를 짚는다
- 가능하면 독자의 혼잣말처럼 시작한다

좋은 시작:

- "이거 그냥 승인 버튼 하나 달면 끝 아닌가 싶을 수 있다."
- "처음 보면 다 비슷해 보여서 뭐가 중요한지 잘 안 잡힌다."

### 3. 예시는 많이, 추상어는 적게

- 추상적 정의만 던지지 않는다
- 실무 장면, 실제 선택, 흔한 실수로 설명한다
- 예시 하나가 정의 세 문단보다 낫다면 예시를 택한다

## 문장 운용 규칙

### 짧게 쓴다

- 한 문장은 가능하면 짧게
- 한 문단은 한 생각만
- 리듬상 강조가 필요하면 한 문장 문단 허용

### 쉬운 말로 쓴다

- 어려운 용어는 처음 등장할 때 바로 푼다
- "그럴듯한 말"보다 "바로 이해되는 말"을 쓴다

예:

- `에스컬레이션`만 쓰지 말고 `상위 검토로 넘기는 것(에스컬레이션)`처럼 푼다

### 능동적으로 쓴다

- "검토가 수행된다"보다 "사람이 검토한다"
- "판단이 요구된다"보다 "판단해야 한다"

## 도입부 규칙

### 제일 좋은 도입: 짧은 장면

다음 네 가지가 보이면 좋다.

1. 누가 있는가
2. 무엇을 하려는가
3. 어디서 틀리거나 헷갈리는가
4. 왜 이게 중요한가

예:

- "금요일 저녁, 고객 메일 자동화가 발송 직전이다. 모델은 환불 문의라고 봤지만 실제론 주소 변경 요청이었다."

### 두 번째로 좋은 도입: 공감 질문

- "이거 다들 좋다는데, 왜 막상 쓰면 답답한지 궁금하지 않았나?"
- "설명은 읽었는데 머릿속에 장면이 안 남는 문서, 은근 많다."

### 피해야 할 도입

- "오늘은 X를 알아보겠습니다"
- "결론부터 말하면"
- 배경설명 3문단 깔고 본론 나오는 방식

## 유머 규칙

유머는 넣어도 된다.
근데 역할이 분명해야 한다.

유머의 목적:

- 독자를 안심시키기
- 어려운 개념의 긴장을 풀기
- 상황을 더 또렷하게 기억하게 만들기

유머의 금지 용도:

- 정보가 빈약한 걸 가리기
- 진지한 판단 포인트를 흐리기
- 문단마다 드립 넣기

좋은 예:

- "승인 단계를 많이 넣는다고 안전해지는 게 아니라, 결재판만 길어진다."
- "설명은 맞는데 머리에 안 남는 글은, 시험 끝나고 증발하는 필기랑 비슷하다."

## 비유 규칙

이 문체에서 비유는 매우 중요하다.
다만 예쁜 비유보다 기억나는 비유를 쓴다.

좋은 비유의 조건:

- 일상적이다
- 바로 그림이 그려진다
- 원래 개념과 1:1로 너무 어긋나지 않는다

추천 패턴:

- 엔진 / 브레이크 / 핸들
- 지도 / 길찾기 / 허브
- 공장 / 흐름 / 병목
- 신호등 / 분기 / 우회

## 문서 유형별 적용

### `index.md`

- 블로그처럼 첫 장면이나 큰 그림으로 열 수 있다
- 하지만 바로 "이 문서에서 뭘 얻는지"를 보여 줘야 한다
- 읽는 순서와 바로가기까지 붙어야 위키 역할을 한다

### `guide`

- 가장 이 문체가 잘 먹히는 문서다
- 장면 -> 원칙 -> 예시 -> 실수 -> 다음 문서 흐름이 좋다

### `concept`

- 정의를 먼저 써도 되지만, 가능하면 "언제 이 개념이 필요해지는가"를 먼저 보여 준다
- 추상 개념일수록 짧은 상황 도입이 효과적이다

### `faq`

- 진짜 사람이 물을 만한 말투를 쓴다
- 너무 교과서 질문처럼 만들지 않는다

## 이 스타일의 체크리스트

- 첫 5문장 안에 독자가 왜 읽어야 하는지 보이는가?
- 한 문단에 한 생각만 있는가?
- 예시나 장면이 추상 설명을 대신해 주는가?
- 유머가 설명을 돕고 있는가, 방해하고 있는가?
- 소제목만 읽어도 흐름이 보이는가?
- 마지막에 기억용 문장 하나는 남는가?

## 금지 사항

- 유튜브 썸네일 같은 과장 문장
- 괜히 센 척하는 문장
- 드립이 본문보다 더 눈에 띄는 문장
- 장면은 있는데 교훈이 없는 글
- 교훈은 있는데 장면이 하나도 없는 글

## 이 프로젝트에서의 실전 적용 방식

1. 구조는 `WRITING-STYLE.md`를 따른다
2. 말맛과 예시 운용은 이 문서를 따른다
3. 추상 문서는 가능하면 `장면 -> 원칙 -> 예시`로 쓴다
4. 고정 문서는 친근하게 쓰되 검색성과 허브 기능을 해치지 않는다

## 한 줄 결론

이 문체는 `재밌게 읽히는 글`이 아니라,
`잘 읽히고, 잘 이해되고, 나중에도 떠오르는 글`을 만들기 위한 문체다.
