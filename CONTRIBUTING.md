# Contributing

이 프로젝트는 "주제만 바꾸면 재사용 가능한 위키 생성 템플릿"을 목표로 한다.
새 기능을 넣을 때도 개별 취향보다 재사용성과 초보자 학습 흐름을 먼저 본다.

## 이슈 등록

- 버그 제보에는 재현 단계, 기대 동작, 실제 동작을 함께 적어 달라.
- 프롬프트 품질 문제라면 어떤 입력에서 어떤 산출물이 어색했는지 예시를 남겨 달라.
- 새 기능 제안은 "누가 왜 필요한지"를 먼저 적어 주면 판단이 빨라진다.

## PR 기준

- 프롬프트를 바꾸면 `.claude/agents/`와 `prompts/`를 함께 맞춰 달라.
- 새 에이전트를 추가하면 README의 에이전트 목록과 관련 예시 설정도 같이 업데이트해 달라.
- 버그 수정 PR은 어떤 상태 파일, 어떤 단계에서 깨졌는지 설명이 있어야 한다.
- 새 `domain_type` 추가 제안은 기존 `tool` / `knowledge`로 커버되지 않는 이유와 IA 변화까지 포함해 달라.
- 문서 예시는 가능한 한 특정 개인 환경에 묶이지 않는 상대 경로를 써 달라.

## 로컬 테스트

검증 산출물은 공개 템플릿 저장소에 커밋하지 않는다.
대신 로컬에서 `validation/` 폴더를 임시 작업장으로 만들어 테스트하면 된다.
이 폴더는 `.gitignore`에 들어 있으니 마음 편히 굴려도 된다.

예시 흐름:

1. `validation/<case-name>/` 폴더를 만든다.
2. 루트 `wiki-config.yaml`과 `wiki-state.json`을 복사하거나 `@wiki-initializer`로 새로 만든다.
3. `output_path`를 `./validation/<case-name>/output` 같은 상대 경로로 맞춘다.
4. `@wiki-orchestrator`를 반복 실행해 문서 생성부터 review까지 끝낸다.
5. 아래 항목을 직접 확인한다.

- `docs_written`, `docs_done`이 기대한 형식으로 유지되는지
- fixed / concept / guide 문서 수가 depth 수준과 맞는지
- `index.md`, `prerequisite-map.md`, `glossary.md`, `faq.md` 링크가 끊기지 않는지
- tool 위키라면 `changelog.md`가 자연스럽게 포함되는지
- publish 전에는 `@wiki-publish-preflight`가 누락 설정을 잘 잡아내는지

## 리뷰 포인트

- 초보자가 읽었을 때 첫 문단만 보고도 "아 뭐 하는 거구나"가 잡히는가
- 선수 지식 링크가 실제 학습 순서를 돕는가
- 같은 이름의 다른 의미가 있을 때 사람 확인 루프가 유지되는가
- 내부 메모 파일이 공개 산출물에 섞이지 않는가

## 커밋 팁

- 하나의 PR에는 하나의 의도만 넣는 편이 좋다.
- README나 예시 설정을 바꿨다면 실제 첫 실행 흐름도 같이 다시 읽어 달라.
- "작동은 하는데 초보자는 헷갈리는" 상태면 아직 완성 아니다. 이 프로젝트는 그 지점을 제일 아프게 본다.
