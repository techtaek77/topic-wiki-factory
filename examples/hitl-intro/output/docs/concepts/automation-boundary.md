# 어디까지 자동화하고 어디서 멈출까

## 먼저 떠올릴 장면

자동화는 잘 돌아간다.
속도도 빠르고 처리량도 많다.
그래서 팀은 점점 더 많은 단계를 "이 정도면 그냥 자동으로 가도 되겠지" 하고 넘기기 시작한다.

문제는 사고가 날 때다.
그 순간 사람들은 묻게 된다.
"이걸 왜 시스템이 혼자 결정하게 뒀지?"

## 자동화 경계가 필요한 이유

자동화는 보통 "할 수 있는가?"만 묻기 쉽다.
하지만 운영에서는 "이걸 시스템이 혼자 해도 되는가?"가 더 중요할 때가 많다.

이 문서에서 기억할 질문은 하나면 된다.

> 할 수 있는가 말고, 혼자 해도 되는가?

## 경계를 정하는 질문

- 이 결정이 고객, 금전, 정책에 직접 영향을 주는가
- 틀렸을 때 되돌리기 쉬운가
- 사람이 보고 바로 수정할 만큼 정보가 충분한가
- 같은 실수가 반복될 때 시스템적으로 막을 수 있는가

## 추천 기본선

- 저위험 / 반복 작업: 자동화 우선
- 고위험 / 애매한 해석: HITL 우선
- 반복적으로 승인되는 패턴: 로그를 모아 점진 자동화 검토

## 흔한 반패턴

- "사람이 마지막에 보면 되지" 하고 초반 근거 수집을 생략한다
- 승인 지연을 고려하지 않고 전체 흐름을 묶는다
- 사람 검토 결과를 학습하지 않아 같은 실수를 계속 반복한다

## 기억에 남는 문장으로 정리하면

자동화 경계는 시스템 능력의 선이 아니라,
사고가 났을 때 사람이 감당할 수 있는 책임의 선이다.
