# Agents and Context

> Codex를 처음 쓸 때 진짜 중요한 건 말빨보다 컨텍스트다. 뭘 할지보다 어디서, 어디까지 할지를 또렷하게 주는 쪽이 훨씬 세다.

---

## 왜 필요한가

보통 사람은 이렇게 생각한다.
"프롬프트만 잘 쓰면 알아서 잘 해주겠지?"
근데 실제로는 시작점이 더 중요하다.

Codex는 지금 열린 작업 폴더와 최근 파일 상태를 바탕으로 움직인다.
그래서 시작점이 애매하면 엉뚱한 파일을 읽거나, 너무 넓게 수정하려 들 수 있다.
질문 센스보다 작업 범위 표지판이 먼저다.

- 없으면: "README 좀 고쳐줘"가 너무 넓게 해석된다.
- 있으면: 작업 폴더, 수정 파일, 금지선이 명확해져 첫 성공 확률이 올라간다.

## 어떻게 적용하는가

처음 요청할 때는 아래 네 줄만 있어도 충분하다.

1. 지금 목표: `README 문장 1개 다듬기`
2. 수정 파일: `README.md`
3. 건드리지 말 것: `코드/설정 파일은 건드리지 않기`
4. 확인 방법: `diff 보여주기`

이 네 줄은 일종의 안전 펜스다.
Codex가 똑똑하더라도, 펜스가 없으면 열심히 멀리 갈 수 있다. 그게 꼭 좋은 방향이란 보장은 없고.

### 첫 요청 예시

```text
이 repo에서 README 첫 문단만 더 명확하게 다듬어줘.
README.md만 수정하고 다른 파일은 건드리지 마.
수정 후에는 어떤 문장이 바뀌었는지 diff 기준으로 설명해줘.
```

### 핵심 포인트

- 처음엔 한 파일, 한 목적만 맡긴다.
- “어디까지 수정해도 되는지”를 같이 준다.
- 결과를 바로 믿기보다 diff를 먼저 본다.

짧게 외우면 이렇다.
`좋은 요청은 멋진 문장보다 작업 범위가 선명한 요청이다.`

### 자주 하는 실수

- 현재 폴더가 엉뚱한데 그냥 시작함 -> 작업할 repo 폴더부터 연다.
- 여러 목적을 한 번에 던짐 -> 구현, 리뷰, 설명을 분리한다.
- "알아서 해줘"만 던짐 -> 목표, 파일, 금지선, 확인 방법을 같이 준다.
