# Review vs Implementation

> 구현 요청과 리뷰 요청은 닮아 보여도 기대하는 출력이 다르다.

---

## 왜 필요한가

구현은 무언가를 만들어내는 일이고, 리뷰는 위험을 찾아내는 일이다.
둘을 섞으면 코드도 애매하고 피드백도 애매해질 수 있다.

- 없으면: "고쳐달라"와 "문제 찾아달라"가 섞여 평가 기준이 흐려진다.
- 있으면: 요청 의도와 결과 형식이 분명해진다.
- 비유: 설계자와 감사팀을 같은 체크리스트로 굴리는 느낌을 피하는 일이다.

## 어떻게 적용하는가

보통 아래처럼 나누면 편하다.

### 구현을 먼저 맡길 때

- 요구사항이 이미 비교적 명확함
- 변경 범위가 작음
- 테스트나 확인 기준이 분명함

### 리뷰를 먼저 맡길 때

- 구조 변경이 큼
- 기존 동작을 깨뜨릴 위험이 큼
- 다른 사람이 만든 diff를 빠르게 검토해야 함

### 핵심 포인트

- 구현 요청에는 목표와 제약을 준다.
- 리뷰 요청에는 위험, 회귀, 빠진 테스트를 우선 보게 한다.
- 둘 다 필요하면 구현 후 리뷰를 나눠서 두 번 호출하는 편이 낫다.
