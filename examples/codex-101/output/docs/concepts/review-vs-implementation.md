# Review vs Implementation

> Codex를 처음 쓸 때 많이 헷갈리는 건 "바로 수정시킬까?" 아니면 "먼저 문제를 찾게 할까?"다.

---

## 왜 필요한가

구현과 리뷰는 요구하는 출력이 다르다.
구현은 파일을 바꾸는 일이고, 리뷰는 위험을 찾는 일이다.
이 둘을 섞으면 결과도 애매해지고 확인 포인트도 흐려진다.

- 없으면: "고쳐줘"와 "문제 찾아줘"가 섞여 기준이 흐려진다.
- 있으면: 지금 뭘 기대해야 하는지가 분명해진다.

---

## 먼저 알아야 할 것

| 개념 | 한 줄 설명 | 링크 |
|------|-----------|------|
| Review | 버그, 회귀, 빠진 테스트를 찾는 검토 단계다. | [Review](../../glossary.md#review) |
| Implementation | 실제 파일을 수정하는 생산 단계다. | [Implementation](../../glossary.md#implementation) |

---

## 어떻게 적용하는가

### 구현부터 시키면 좋은 경우

- 수정 파일이 1~2개 정도로 작다
- 기대 결과가 비교적 분명하다
- 나중에 diff를 직접 확인할 수 있다

### 리뷰부터 시키면 좋은 경우

- 다른 사람이 만든 변경을 빠르게 읽어야 한다
- 회귀 위험이 큰 수정이다
- 당장 코드보다 "뭐가 위험한지"가 더 궁금하다

### 첫 요청 예시

구현:

```text
README.md 첫 문단을 더 명확하게 다듬어줘.
문서 톤은 유지하고 다른 파일은 건드리지 마.
```

리뷰:

```text
이 브랜치 변경을 리뷰해줘.
버그, 회귀 위험, 빠진 테스트를 우선순위대로 알려줘.
```

### 핵심 포인트

- 첫 작업은 구현/리뷰를 섞지 않는다.
- 구현 후에는 diff를 보고 필요하면 리뷰를 한 번 더 건다.
- 리뷰 요청에는 "뭘 우선으로 봐야 하는지"를 같이 준다.
