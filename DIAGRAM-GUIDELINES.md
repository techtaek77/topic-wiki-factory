# Diagram Guidelines

이 프로젝트는 "설명하기 어려운 것을 그림으로 바꾸는 것"보다
"이해시키고 기억시키고 싶은 구조를 그림으로 고정하는 것"을 목표로 한다.

차트나 도해는 장식이 아니다.
글보다 빨리 이해시키거나, 글만으로는 놓치기 쉬운 구조를 한 번에 잡게 해야 한다.

## 언제 넣는가

아래 중 하나라도 해당되면 다이어그램 후보로 본다.

- 단계가 3개 이상 이어지는 흐름을 설명한다
- 분기가 2개 이상 생긴다
- 문서나 개념 사이 의존성이 3개 이상 있다
- before / after 차이를 보여 주는 편이 훨씬 빠르다
- 공간 배치, 이동 경로, 상태 변화가 핵심이다
- 텍스트로 3문단 넘게 설명해야 겨우 이해되는 구조다

반대로 아래라면 굳이 넣지 않는다.

- 문장 2~3개면 이미 충분히 선명하다
- 그림이 내용을 줄이는 게 아니라 되풀이만 한다
- 예뻐 보이기만 하고 의사결정이나 이해 속도를 올려 주지 않는다

## 어떤 형식을 쓰는가

### 1. `mermaid flowchart`

용도:

- 단계 흐름
- 승인/거절 분기
- 상태 전이
- 시작 -> 판단 -> 다음 액션 구조

좋은 예:

- HITL 승인 흐름
- orchestrator phase 전이
- quick-start 루프

### 2. `mermaid graph`

용도:

- 선수지식 맵
- `priority + depends_on`
- 개념 간 관계
- 문서 간 참조 구조

좋은 예:

- basics -> glossary -> quick-start -> guide 연결
- fixed 문서와 concept/guide 의존 관계

### 3. `SVG` 또는 정적 이미지

용도:

- 체스판 배치
- UI before / after
- 위치, 각도, 이동 경로
- 텍스트보다 그림이 훨씬 빠른 규칙 설명

좋은 예:

- castling
- en passant
- approval screen layout

## 기본 원칙

### 차트는 본문보다 먼저 뽐내지 않는다

- 차트 앞에 1~3문장 정도로 "이 그림이 뭘 보여 주는지"를 말한다
- 차트 뒤에는 꼭 읽는 포인트를 짧게 짚는다
- 그림만 던지고 끝내지 않는다

### 차트는 한 장에 한 생각

- 하나의 차트에 단계 흐름, 책임 분리, 예외 규칙을 다 욱여넣지 않는다
- 필요한 경우 차트를 둘로 나눈다

### 차트의 라벨은 설명형으로 쓴다

- "Step A", "Node 1" 같은 라벨은 피한다
- 노드만 읽어도 흐름이 보이게 쓴다

좋은 예:

- `고위험이거나 애매한가?`
- `사람 검토 요청`
- `초안 작성 후 diff 확인`

## 문서 유형별 권장

### `index.md`

- 큰 흐름을 한 장으로 요약하는 차트가 있으면 좋다
- 특히 시스템/워크플로우 주제라면 `flowchart`가 잘 맞는다

### `guide`

- 따라 하는 순서가 핵심이면 `flowchart`를 우선 고려한다
- 실무 루프나 체크 순서는 차트가 특히 잘 먹힌다

### `concept`

- 관계망이 핵심이면 `graph`
- 상태 변화나 판단 분기면 `flowchart`

### `prerequisite-map.md`

- 선수지식 구조가 핵심이면 `graph`를 우선 고려한다

## 리뷰 기준

좋은 차트는 아래를 만족한다.

- 차트가 없어도 문서는 읽히지만, 차트가 있으면 훨씬 빨리 이해된다
- 차트 제목이나 앞문장만 읽어도 용도가 보인다
- 노드와 화살표가 실제 문장 구조와 충돌하지 않는다
- 차트가 본문을 요약하거나 압축하지, 복붙하지 않는다

## 한 줄 결론

차트는 `예뻐 보이는 장식`이 아니라,
`독자의 머릿속에 구조를 한 번에 박아 넣는 압축 도구`다.
